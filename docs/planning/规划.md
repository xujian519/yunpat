通过对 `DeepSeek-TUI` 仓库（xujian519/DeepSeek-TUI，基于 Hmbown 的上游版本）的代码结构、文档（`ARCHITECTURE.md`、`AGENTS.md`）以及最新更新日志的分析，该项目目前正处于从“终端聊天工具”向“全功能 Agent 编码助手”转型的关键期。

如果你要实现**宪法规划（系统性架构规划与核心准则确立）**，下一步需要重点推进以下四个维度的演进：

### 1. 核心架构：由“单体 TUI”向“解耦引擎”彻底演进

目前项目虽然有 `crates/` 目录，但大部分逻辑仍耦合在 `crates/tui` 中。

- **行动点：** 将 **Agent Engine**（推理循环、工具调度、状态管理）彻底抽离为独立 crate，使其不依赖于 Ratatui。
- **目标：** 实现“一核多端”。核心引擎可以同时支持当前的 TUI、Headless 模式（用于 CI/CD 或自动化工作流 `deepseek serve --http`）以及未来的 IDE 插件（如 VS Code 扩展）。

### 2. 内存与上下文治理：建立“分级记忆宪法”

当前文档提到长会话会导致崩溃或性能下降，且自动压缩（Compaction）机制尚不稳定。

- **行动点：** 实现**多级上下文管理机制**：
  - **短期记忆：** 当前任务的原始对话流。
  - **中期记忆：** 基于 RLM（Recursive Language Model）的自动摘要，保存已完成子任务的结论而非过程。
  - **长期记忆：** 强化 `~/.deepseek/instructions.md` 和用户偏好文件（User Memory），实现跨会话的知识沉淀。
- **目标：** 解决 1M Token 背景下的“上下文膨胀”问题，提高前缀缓存（Prefix Cache）命中率以降低成本。

### 3. 工具宪章：标准化 MCP 与沙盒策略

项目已引入 MCP（Model Context Protocol），但工具执行的安全边界（Sandbox）仍需加固。

- **行动点：**
  - **强化权限模型：** 细化 `DEEPSEEK_SANDBOX_MODE`，建立类似移动端 App 的权限申请机制（例如：访问 workspace 以外的目录需显式授权）。
  - **工具元数据标准化：** 为所有内置工具（shell, git, file_ops）编写严格的 JSON Schema 和验证逻辑，防止 LLM 生成幻觉参数导致系统风险。

### 4. 协作协议：完善“子代理（Sub-agent）”调度逻辑

`AGENTS.md` 建议用户手动生成子代理，这增加了用户的心智负担。

- **行动点：** 开发**自动拆解规划器（Auto-Planner）**。当任务复杂度超过阈值时，引擎应能自动执行 `agent_spawn` 拆分任务，并在主线程进行结果汇聚。
- **目标：** 实现从“用户指挥工具”到“用户指挥组长（Coordinator），组长指挥工人（Workers）”的模式转变。

### 推荐下一步具体操作建议：

1. **代码层面：** 优先重构 `core/engine/turn_loop.rs`，将其逻辑标准化，以便支持异步并发处理多个工具调用。
2. **配置层面：** 引入 `deepseek.toml` 集中管理所有“宪法级”配置项（如默认模型、费率限制、默认安全级别）。
3. **诊断层面：** 增强 `deepseek doctor` 功能，使其不仅能检查 API 连接，还能检测本地 LSP 服务和 MCP 服务器的健康度。

结合您在专利领域的深厚背景（专利代理师、知识产权战略研究员）以及 `DeepSeek-TUI`（高效终端 Agent 引擎）与 `yunpat`（基于知识图谱的专利法律库）的开源现状，这两个项目的深度融合将创造一个“AI 驱动的知识产权专家工作站”。

以下是为您规划的长期发展路径，分为三个阶段，旨在将法律知识图谱的“专业性”**与 AI Agent 的**“执行力”完美结合：

### 阶段一：专业对齐与检索增强（RAG 2.0 阶段）

**核心目标：** 让 DeepSeek-TUI 具备“专家级”的法律直觉，解决通用模型对中国专利法理解不深的问题。

- **知识库无缝接入：**
  - 将 `yunpat` 中的 3,000 部法律和 11,000 份无效决定 转化为 DeepSeek-TUI 的**本地向量索引（Vector DB）**。
  - 开发 `yunpat-mcp` 插件，使 TUI 能直接通过 MCP 协议调用专利检索 API，实现从“对话”到“查法条/查案例”的瞬间切换。
- **专家级 System Prompt 宪法：**
  - 利用您在专利无效和行政诉讼方面的经验，为 TUI 定制“专利无效分析”专用模式（Agent Mode）。
  - **下一步行动：** 在 `~/.deepseek/instructions.md` 中固化无效请求书的撰写准则和对比文件审查逻辑。

### 阶段二：自动化专家工作流（Agentic Workflow 阶段）

**核心目标：** 将 TUI 的 Agent 能力应用到专利实务场景，实现从单点问答到“任务闭环”。

- **专利无效分析 Agent：**
  - **自动拆解：** 输入一份专利号，Agent 自动调用工具下载权利要求书、说明书及引证文件。
  - **特征比对：** 模仿您之前进行的“双氧化层生物质气化炉”技术特征分析逻辑，由子代理（Sub-agent）自动拆分权利要求特征点，并与对比文件进行三步法（或五步法）比对。
- **自动化自媒体运营集成：**
  - 结合您正在构建的“智能自媒体运营系统”，让 TUI 能够自动提取最新的专利判例（来自 `yunpat`），撰写专业评论文章，并自动发布。

### 阶段三：云端与本地协同的“IP 大脑”（Ecosystem 阶段）

**核心目标：** 打造一个私密、高效、可扩展的知识产权数字化身。

- **数据隐私与本地化：**
  - 响应您对数据隐私的关注，实现关键法律知识图谱完全本地化存储（如使用 DuckDB 或嵌入式图数据库）。
  - 支持本地 LLM（如通过 Ollama 运行 DeepSeek-R1 本地版）处理敏感案卷。
- **多端联动：**
  - 将 TUI 的核心引擎（Crate）封装，支持您计划中的移动端小程序，让专利代理人在法庭或客户端现场也能通过语音输入直接检索 `yunpat` 库并生成法律意见。

### 核心价值导向

- **对您：** 释放重复劳动。TUI 负责处理 80% 的检索、格式化比对和初稿撰写；您负责 20% 的核心策略和决定性判断。
- **对行业：** `yunpat` 提供了高质量的结构化数据，`DeepSeek-TUI` 提供了现代化的交互界面。二者结合是法律科技（LegalTech）从“数字化”向“智能化”跨越的典型样本。
