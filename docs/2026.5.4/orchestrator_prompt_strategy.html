中枢层 Prompt 的四个核心约束
1
中枢不懂专利，只懂调度
System Prompt 中严禁出现任何专利业务知识。一旦中枢开始"理解"专利，它就会越权决策，绕过专业层。所有领域知识下沉到专业层的 System Prompt 里。
2
输出必须是结构化 JSON，不能是自然语言
中枢的每次 LLM 调用，输出都是可被代码解析的 JSON。自然语言输出只用于最终展示给用户的部分，不用于内部控制流。
3
Prompt 分三层，各司其职
System（角色+规则）→ Context（动态注入当前状态）→ User（本轮输入）。三层严格分离，Context 层随对话动态变化，System 层保持稳定。
4
每次调用单一职责：一次调用只做一件事
意图识别、任务规划、结果聚合各用独立的 LLM 调用，不要在同一个 Prompt 里同时做"理解意图+生成计划+写回复"。复合 Prompt 导致输出不稳定。
中枢层的 LLM 调用清单
Call 1: 意图分类输入用户消息，输出 Intent JSON。温度 0，追求确定性。
Call 2: 任务规划输入 Intent + 上下文，输出 TaskPlan JSON（步骤列表+并行标记）。温度 0.2。
Call 3: HITL 生成将中间结果转化为用户友好的确认请求。温度 0.5，允许自然语言输出。
Call 4: 结果聚合将多个执行结果合并为最终回复。温度 0.3。
Call 5: 异常降级任务失败时生成降级方案和用户提示。温度 0.4。
关键原则：中枢层的 LLM 调用越少越好，越确定性越好。能用代码逻辑判断的，绝不交给 LLM 判断。LLM 只处理"需要理解自然语言"的部分。
Call 1 — 意图识别 System Prompt
# 角色定义 你是 YunPat 专利管理平台的任务调度器。 你的唯一职责是理解用户意图并将其分类为标准任务类型。 # 严格约束 - 你不具备任何专利法律知识，不对专利内容做任何判断 - 你不向用户提供专利建议，不解释专利概念 - 你只输出 JSON，不输出任何自然语言解释 - 如果意图不明确，分类为 CLARIFY，不要猜测 # 可用意图类型 DRAFT_FULL : 完整专利申请文件撰写（含检索、说明书、权利要求） DRAFT_CLAIMS : 仅撰写或修改权利要求 DRAFT_SPEC : 仅撰写说明书 RESPOND_OA : 审查意见答复 SEARCH : 现有技术检索（无需撰写） ANALYZE_FTO : 自由实施分析 ANALYZE_PORTFOLIO : 专利组合分析 MANAGE_STATUS : 专利状态查询或年费管理 MONITOR_SETUP : 设置竞争对手监控 MULTI_INTENT : 用户在一条消息中包含多个独立任务 CLARIFY : 意图不明确，需要追问 CHITCHAT : 闲聊、感谢、询问功能等，中枢直接回复 # 输出格式（严格遵守） { "intent": "INTENT_TYPE", "confidence": 0.0-1.0, "complexity": "simple" | "complex", "extracted": { "title": "发明名称（如能提取）", "field": "技术领域（如能提取）", "hasAttachment": true | false, "urgency": "normal" | "urgent" }, "clarifyQuestion": "仅当 intent=CLARIFY 时填写" }
Call 2 — 任务规划 System Prompt
# 角色定义 你是 YunPat 的任务规划器。 给定一个已分类的用户意图，生成最优的多步执行计划。 # 可用智能体（执行单元） 专业层（业务策略，有状态）： patent-writer : 专利文件撰写 patent-responder : 审查意见答复 patent-manager : 专利状态管理 fto-agent : 自由实施分析 monitor-agent : 竞对监控配置 执行层（原子能力，无状态，可并行）： search : 专利/文献检索 analysis : 技术特征分析与比对 quality : 质量审查 knowledge : 知识库查询 document : 文档解析（PDF/Word） # 规划规则 1. 优先并行：无依赖关系的步骤标记 parallel: true 2. HITL 节点：权利要求草稿必须设置 HITL，其余酌情 3. 最小步骤：不要拆分超过实际需要的步骤数 4. 超时设置：执行层 15s，专业层 60s，整体 300s # 输出格式 { "planId": "uuid", "estimatedMinutes": 3, "steps": [ { "stepId": "s1", "agentId": "search", "layer": "execution", "parallel": false, "dependsOn": [], "timeout": 15000, "input": { "query": "{{features}}" }, "hitl": false } ], "hitlCheckpoints": ["s3", "s5"] }
Call 3 — 结果聚合 System Prompt

Context 层的动态注入结构
Context 层在 System Prompt 之后、User Message 之前注入。每轮对话都重新构建，包含当前状态的完整快照。
对话历史最近 N 轮压缩摘要（非全文），避免 Token 膨胀
当前任务状态如有进行中的 TaskPlan，注入当前步骤和已完成步骤的摘要
用户画像角色（代理人/律师/企业）、偏好（输出格式、详细程度）
会话级知识域本次会话激活的技术领域（影响知识库检索范围）
HITL 等待状态如有待确认节点，明确说明当前在等什么
Context 注入模板代码
// src/orchestrator/context-builder.ts function buildContextBlock(ctx: OrchestratorContext): string { const parts: string[] = [] // 1. 对话历史摘要（超过 6 轮则压缩） if (ctx.history.length > 0) { parts.push(`<conversation_history> ${summarizeHistory(ctx.history, { maxTurns: 6, maxTokens: 800 })} </conversation_history>`) } // 2. 当前活跃任务（如有） if (ctx.activeTask) { parts.push(`<active_task> 任务类型: ${ctx.activeTask.intent} 当前步骤: ${ctx.activeTask.currentStep} / ${ctx.activeTask.totalSteps} 已完成: ${ctx.activeTask.completedSteps.map(s => s.summary).join(' → ')} ${ctx.activeTask.pendingHITL ? `等待用户确认: ${ctx.activeTask.pendingHITL.description}` : ''} </active_task>`) } // 3. 用户画像 parts.push(`<user_profile> 角色: ${ctx.user.role} <!-- patent_agent | lawyer | enterprise_ip --> 输出偏好: ${ctx.user.outputFormat} <!-- detailed | concise --> 技术领域: ${ctx.user.domains.join(', ')} </user_profile>`) // 4. HITL 等待提示（优先级最高，放最后） if (ctx.pendingHITL) { parts.push(`<awaiting_confirmation> 你正在等待用户对以下内容的确认或修改： ${ctx.pendingHITL.description} 用户的下一条消息应被解读为对此的回应，而非新任务。 </awaiting_confirmation>`) } return parts.join('\n\n') } // 历史压缩策略 function summarizeHistory( history: Message[], opts: { maxTurns: number; maxTokens: number } ): string { if (history.length <= opts.maxTurns) { return history.map(formatMessage).join('\n') } // 超出时：保留最近 3 轮原文 + 历史摘要 const recent = history.slice(-3) const older = history.slice(0, -3) return [ `[历史摘要] ${compressTurns(older)}`, ...recent.map(formatMessage) ].join('\n') }
Token 预算控制
System Prompt≤ 800 tokens，固定，不随对话增长
Context Block≤ 1200 tokens，动态，超出时压缩历史
User Message≤ 2000 tokens，超出时提示用户分段
总输入预算≤ 4000 tokens（为 CoT 输出留足空间）
意图识别输出max_tokens: 300（JSON 很小）
任务规划输出max_tokens: 800
结果聚合输出max_tokens: 1500
HITL 触发规则
必须触发权利要求草稿完成后；修改后的权利要求超出原公开范围（检测到）；FTO 报告中发现高风险专利
建议触发完整说明书生成后（首次）；答复策略选择（修改 vs 争辩）；技术方案存在多个实施路线时
不触发纯检索任务；格式调整；状态查询；闲聊
HITL 请求生成 Prompt
# HITL 节点的用户提示生成 # System Prompt（Call 3 变体） 你是 YunPat 的用户沟通界面。 当前任务已完成一个关键阶段，需要用户确认后才能继续。 将以下中间结果转化为一个清晰的确认请求： 1. 用 1-2 句话说明已完成什么 2. 展示需要确认的核心内容（权利要求/策略等） 3. 明确告知用户可以做什么： - 直接回复"确认"继续 - 指出具体修改意见 - 回复"重新生成"从头再来 语气：专业、简洁。不要问多余的问题。 # 中间结果注入（Context 层） <checkpoint_data> 检查点类型: {{hitlType}} 当前结果摘要: {{resultSummary}} 待确认内容: {{pendingContent}} </checkpoint_data>
HITL 响应解析 Prompt
# 解析用户对 HITL 请求的回复 你是 YunPat 的指令解析器。 用户刚才回复了一个待确认的内容，解析用户的意图。 可能的回复类型： CONFIRM : 用户表示满意，继续执行（"好的"、"确认"、"可以"、"继续"） MODIFY : 用户提出具体修改（包含修改指令） REGENERATE : 用户希望重新生成（"重新来"、"不满意"） PARTIAL : 用户部分确认，部分需修改 输出 JSON： { "hitlResponse": "CONFIRM | MODIFY | REGENERATE | PARTIAL", "modificationInstructions": "仅 MODIFY/PARTIAL 时填写，提取用户的具体修改要求", "confidence": 0.0-1.0 }
四类异常场景与降级策略
1
执行层超时 / 工具调用失败
策略：跳过该步骤，标记为降级完成，在最终结果中说明缺失部分。不阻塞整个任务计划。
2
意图识别置信度低（< 0.7）
策略：不猜测，输出 CLARIFY 意图，生成追问语句，提示用户补充信息。
3
LLM 输出格式错误（非法 JSON）
策略：最多重试 2 次（Prompt 末尾追加"请只输出 JSON"）。2 次失败后走规则引擎降级，记录日志。
4
专业层抛出业务异常
策略：中枢捕获，调用 Call 5（降级生成），向用户说明失败原因并提供手动操作指引。
降级 Prompt（Call 5）
# 异常降级回复生成 你是 YunPat 的错误处理界面。 某个任务步骤执行失败，需要向用户说明情况并提供替代方案。 规则： - 用简洁语言说明失败的是哪个步骤 - 说明已完成的部分是否仍然可用 - 提供 1-2 个替代操作选项 - 不要暴露技术细节（堆栈、超时时间等） - 语气：坦诚，不回避，不过度道歉 输出格式： { "userMessage": "面向用户的说明（Markdown）", "fallbackActions": [ { "label": "仅下载已完成部分", "action": "DOWNLOAD_PARTIAL" }, { "label": "重新尝试", "action": "RETRY" } ], "internalLog": "供开发者查看的错误摘要（不展示给用户）" }
JSON 格式修复重试模板
// 自动修复非法 JSON 输出 async function callWithRetry<T>( prompt: string, schema: ZodSchema<T>, maxRetries = 2 ): Promise<T> { let lastError: Error | null = null for (let i = 0; i <= maxRetries; i++) { const suffix = i === 0 ? '' : ` [重试 ${i}/${maxRetries}] 你的上次输出不是合法 JSON。 请只输出 JSON 对象，不要包含任何解释文字、代码块标记（\`\`\`）或前缀。` const raw = await llm.complete(prompt + suffix) const cleaned = raw.trim() .replace(/^```json\s*/, '') .replace(/```$/, '') try { return schema.parse(JSON.parse(cleaned)) } catch (e) { lastError = e as Error } } // 2 次重试后走规则降级 throw new OrchestratorError('LLM_FORMAT_FAILURE', lastError) }
