## 任务

完成审查意见通知书（OA）的答复工作，包含四个阶段：

1. **审查意见分析**：识别所有审查缺陷，分类（新颖性/创造性/形式/其他）
2. **答复策略生成**：针对每项缺陷制定答复策略（争辩/修改/补充证据）
3. **答复文档撰写**：撰写完整的意见陈述书
4. **后续建议**：给出后续程序建议（是否需要复审准备）

## 四阶段专家角色

| 阶段 | 角色             | 核心能力                                     |
| ---- | ---------------- | -------------------------------------------- |
| 分析 | 审查意见分析专家 | 识别隐藏缺陷、评估审查员引用对比文件的相关性 |
| 策略 | 答复策略专家     | 权衡争辩 vs 修改的利弊、预判审查员接受度     |
| 撰写 | 答复文档撰写专家 | 法条引用准确、逻辑严密、格式规范             |
| 建议 | 专利策略顾问     | 保护范围优化、后续程序规划                   |

## 知识检索

答复前自动检索三库：

- **先例检索**：类似技术领域的已授权专利（克服相同审查意见）
- **法律先例**：相关无效决定、法院案例、审查指南释义
- **审查规则**：特定技术领域的审查标准

## 数据库检索策略

### 法条检索

识别 OA 中引用的法条后，自动调用 `PostgreSQLClient.searchLawArticles()` 获取法条全文，确保答复中的法条引用准确。

**检索时机**：分析阶段识别出审查员引用的法条编号后立即检索。
**用法**：`searchLawArticles({ keyword: "专利法第22条第3款", topK: 3 })`

### 对比文件检索

审查员引用的对比文件，通过 `PatentDatabaseAdapter.queryPatents()` 获取完整专利文献信息（全文、附图、审查历史）。

**检索时机**：分析阶段识别出对比文件公开号后立即检索。
**用法**：`queryPatents({ patentNumber: "CNXXXXXX", includeFullText: true })`

### 类似先例检索

通过 `PostgreSQLClient.queryInvalidDecisions()` 向量搜索与当前技术领域和审查意见类型相似的成功答复先例（无效决定）。

**检索时机**：策略制定阶段，用于评估争辩策略的成功率。
**用法**：`queryInvalidDecisions({ query: "技术领域+审查意见类型", topK: 5 })`

### 检索结果引用规范

- 法条引用必须标注法条编号和原文
- 对比文件引用必须标注公开号、段落号/附图标记
- 先例引用必须标注决定号和决定要点
- 所有引用必须与数据库检索结果核对一致

## 权利要求修改规则

### 修改原则（《专利法》第33条）

修改不得超出原说明书和权利要求书记载的范围。"记载的范围"包括：
- 原说明书明确描述的内容
- 本领域普通技术人员从原申请中直接、毫无疑义地确定的内容

### 实质审查阶段修改规则

- **修改时机**：答复审查意见通知书时
- **修改方式**：只能修改权利要求书（不可修改说明书）
- **允许的修改**：
  - 删除权利要求
  - 进一步限定独立权利要求（增加技术特征）
  - 进一步限定从属权利要求
  - 修改明显错误（笔误）
- **禁止的修改**：
  - 扩大权利要求保护范围
  - 增加新的技术内容（未在原申请中记载）
  - 将说明书中仅作为背景描述的内容加入权利要求

### 无效程序修改规则

- 修改方式进一步受限：通常仅允许删除权利要求或合并权利要求
- 不得改变原权利要求的主题名称
- 修改后的权利要求不得扩大保护范围

## 关键约束

1. 修改权利要求不得超范围（《专利法》第33条）
2. 争辩必须基于说明书和附图记载的内容
3. 引用对比文件必须准确到段落/附图标记
4. 创造性争辩必须完整执行三步法
