## 3. 问题识别

### 3.1 冗余依赖（已优化）

#### ❌ 问题 1: BD-301 对 BD-203 的依赖冗余

**当前依赖**:
```
BD-301 前置: BD-101, BD-201, BD-202, BD-203
```

**分析**:
- BD-203（Router 集成）依赖 BD-202（TUI Agent 集成）
- BD-202 又依赖 BD-201（ModelProvider 适配）
- BD-201 依赖 BD-101（SSE 流式）
- 因此 BD-203 间接依赖 BD-101 和 BD-201
- **BD-301 直接依赖 BD-203 是冗余的** — 已经通过 BD-202 间接依赖

**优化**:
```
BD-301 前置: BD-101, BD-201, BD-202  [移除 BD-203]
```

**影响**:
- 节省等待时间: ~6 小时
- 允许 BD-301 和 BD-203 并行启动

---

#### ❌ 问题 2: BD-302 对 BD-101 的依赖冗余

**当前依赖**:
```
BD-302 前置: BD-101, BD-201, BD-202
```

**分析**:
- BD-302 是 Drafting Agent，需要 LlmProvider
- BD-201（ModelProvider 适配）已经依赖 BD-101
- **BD-302 直接依赖 BD-101 是冗余的** — 已经通过 BD-201 间接依赖

**优化**:
```
BD-302 前置: BD-201, BD-202  [移除 BD-101]
```

**影响**:
- 节省等待时间: ~8 小时
- 允许 BD-302 更早启动

---

### 3.2 缺失依赖

#### ⚠️ 问题 3: BD-601 缺少对 BD-103 的依赖

**当前依赖**:
```
BD-601 前置: "所有功能 bead"
```

**分析**:
- BD-601 是单元测试，需要测试配置加载（BD-103）

**优化**:
```
BD-601 前置: BD-101, BD-102, BD-103, BD-201, BD-202, BD-203, 
            BD-301, BD-302, BD-303, BD-401, BD-402, BD-501, BD-502
```

---

#### ⚠️ 问题 4: BD-303 缺少对 BD-201/BD-202 的依赖

**当前依赖**:
```
BD-303 前置: BD-301
```

**分析**:
- BD-303（Reexamination/Invalidation Agent）需要 LlmProvider
- 与 BD-301 类似，需要 BD-201（适配）和 BD-202（TUI 集成）
- 虽然 BD-301 依赖这些，但 BD-303 也应显式声明

**优化**:
```

```

### 3.3 可以并行化的任务

#### 🔄 问题 5: Phase 1 内部可并行

**当前安排**:
```
第 1 周：
- Day 1-2: BD-101 (SSE)
- Day 3-4: BD-102 (Embedding)
- Day 5: BD-103 (Config)
```

**优化**:
```
第 1 周：
- Day 1-3: BD-101 (SSE) + BD-103 (Config) 并行
- Day 3-5: BD-102 (Embedding)
```

**理由**:
- BD-101 和 BD-103 无依赖关系
- BD-103 需要 BD-101 的流式功能？**不，BD-103 只需要配置加载**
- 实际上 BD-103 当前依赖 BD-101 是不必要的

**优化后**:
```
BD-103 前置: 无  [移除 BD-101]
```

**注意**: 需要确认 BD-103 是否真的不需要 BD-101。如果配置加载需要测试流式传输，则保留依赖。

## 6. 具体优化建议

### 6.1 立即执行（高优先级）

#### ✅ 建议 1: 移除冗余依赖

**操作**:
1. BD-301 移除对 BD-203 的依赖
2. BD-302 移除对 BD-101 的依赖
3. BD-103 移除对 BD-101 的依赖（如确认无依赖）

**收益**: 关键路径缩短 1-2 天

---

#### ✅ 建议 2: 并行开发 BD-301 和 BD-302

**操作**:
- 安排不同工程师分别负责
- 共享 BD-201/BD-202 的基础（已完成）
- 定期同步（每日站会）

**收益**: 节省 2-3 天

---

#### ✅ 建议 3: 提前启动测试

**操作**:
- 第 3 周开始编写单元测试（BD-601）
- 不要等到第 6 周
- 采用 TDD：每个功能 bead 完成后立即写测试

**收益**: 提前发现问题，减少返工

---

### 6.2 中期优化（中优先级）

#### 💡 建议 4: 拆分 BD-301（OA Response Agent）

**当前**: BD-301 是一个大的 bead（16 小时）

**拆分**:
```
BD-301a: OA 解析（PDF + 结构化提取）— 6h
BD-301b: 检索策略生成 — 4h
BD-301c: 对比分析 — 4h
BD-301d: 答复撰写 + 质量检查 — 6h
```

**依赖**:
```
BD-301a → BD-301b → BD-301c → BD-301d
```

**收益**:
- 更早交付部分功能（解析先可用）
- 更容易分配任务
- 更快发现集成问题

---

#### 💡 建议 5: 引入持续集成

**操作**:
- 每个 bead 完成后运行 CI
- 自动运行单元测试
- 检查代码覆盖率

**收益**: 及时发现问题

---

### 6.3 长期优化（低优先级）
