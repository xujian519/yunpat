# DeepSeek-TUI × YunPat 融合项目深度分析报告

> 分析日期：2026-05-08
> 分析目标：制定长期融合规划，包括代码库URL/仓库策略

---

## 一、两个项目的本质画像

### 1.1 DeepSeek-TUI（执行层王者）

| 维度 | 详情 |
|------|------|
| **定位** | 通用编码Agent/TUI框架 |
| **语言** | Rust（100%） |
| **规模** | 21个crate，369个Rust源文件 |
| **核心能力** | Agent编排、工具调度、Session管理、MCP协议、RLM并行推理 |
| **架构** | 分层Workspace（cli/tui/app-server → core → tools/mcp/hooks） |
| **TUI** | ratatui（Rust原生，生产就绪） |
| **模式** | Plan/Agent/YOLO三种运行模式 |
| **安全** | execpolicy执行策略、沙箱机制、审批门控 |
| **短板** | 无任何领域知识，面对专业任务（专利/法律）完全空白 |

**核心价值**：**执行层的极致成熟度** —— session恢复、workspace回滚、LSP诊断、cost tracking、多语言本地化、sub-agent生命周期管理。

### 1.2 YunPat（专利业务层专家）

| 维度 | 详情 |
|------|------|
| **定位** | 知识产权全生命周期智能体平台 |
| **语言** | TypeScript（70%）+ Rust（30%）+ Python（隔离） |
| **规模** | 21个顶层包 + 31个agent子包，489个TS源文件，~63,000行核心代码 |
| **核心能力** | 专利检索（7500万CN专利）、权利要求生成、审查答复、可专利性分析 |
| **架构** | 五层架构（Gateway→Reasoning→LLM→Memory→Tools） |
| **知识库** | 4,382个markdown文件，覆盖专利法/审查指南/判例 |
| **数据库** | PostgreSQL（7500万CN专利）+ Google Patents API |
| **Agent** | 29个专业Agent（撰写/检索/分析/答复/质量检查等） |
| **TUI** | React/Ink（TypeScript，80%完成，计划被替代） |
| **短板** | 执行层薄弱，TUI才80%，无session恢复等核心能力 |

**核心价值**：**专利领域的深度专业知识** —— 从检索到撰写到答复的完整工作流，4382个知识库文件构成的领域壁垒。

### 1.3 融合的本质

> **不是"1+1=2"的项目合并，而是"大脑+躯体"的深度融合。**

- **YunPat的专利大脑** + **DeepSeek-TUI的执行躯体** = **专利宪法级智能体操作系统**
- YunPat提供：领域知识、专业Agent、规则引擎、专利数据库
- DeepSeek-TUI提供：终端交互、session管理、工具调度、sub-agent编排、MCP协议

## 二、架构差异与融合挑战

### 2.1 技术栈异构（最大挑战）

| 维度 | DeepSeek-TUI | YunPat | 融合策略 |
|------|-------------|--------|----------|
| **主要语言** | Rust | TypeScript | **保持双语言**，通过MCP/gRPC建立边界 |
| **包管理** | Cargo Workspace | pnpm Workspace | 各自独立，通过协议通信 |
| **TUI框架** | ratatui（Rust） | Ink（React/TS） | **废弃Ink，统一ratatui** |
| **构建工具** | cargo | pnpm/esbuild | 各自独立 |
| **运行时** | Rust原生 | Node.js 18+ | 各自独立 |

**关键决策**：**不要试图把TypeScript业务逻辑重写成Rust**。YunPat的6万行业务代码（29个专业Agent、4382个知识库文件）重写成本极高且风险巨大。正确的做法是**建立稳定的跨语言边界**。

### 2.2 架构哲学差异

| 维度 | DeepSeek-TUI | YunPat |
|------|-------------|--------|
| **Agent循环** | 扁平ReAct循环（LLM→tool→LLM→tool） | 分层规划-执行架构（意图识别→任务分解→子Agent路由→执行→异常处理） |
| **任务调度** | 模型自主决定工具调用 | Orchestrator集中式调度 |
| **记忆管理** | SQLite持久化 + 上下文压缩 | 五层记忆（短期/长期/检查点/断点续传/时间旅行） |
| **安全模型** | execpolicy审批门控 | Gateway认证 + HITL人机协作 |

**融合策略**：**不是替换，而是分层叠加** —— 把YunPat的Orchestrator作为"规划前置层"叠加在DeepSeek-TUI的Engine循环之上。

### 2.3 数据主权与合规（专利场景核心）

专利场景对数据安全的要求远超通用编码：
- **技术交底书**属于企业核心机密，一旦泄露可能造成数亿损失
- **审查答复策略**暴露给客户或竞争对手会影响专利价值
- **权利要求措辞**的修改历史是重要的法律证据

**当前差距**：
- DeepSeek-TUI有沙箱和审批门控，但无**数据主权强制约束**
- YunPat有"本地优先承诺"，但执行层无强制保证

## 三、长期融合路线图（四阶段）

### Phase 1：接口对齐（0-3个月）

**目标**：让DeepSeek-TUI作为YunPat的终端执行层直接可用，最小侵入，快速打通。

| 任务 | 工作量 | 关键决策 |
|------|--------|----------|
| **MCP协议对接** | 1-2周 | YunPat的`packages/mcp-server`（6个源文件）通过MCP stdio暴露专利Agent为工具 |
| **Skills格式统一** | 2-3周 | YunPat的TypeScript Skills导出为DeepSeek-TUI兼容的`SKILL.md`格式 |
| **TUI层取舍** | 1周 | **废弃YunPat Ink TUI**，精力集中在Rust TUI的专利场景定制 |
| **项目级配置** | 3天 | 在YunPat项目根目录创建`.deepseek/config.toml`和专利系统指令 |

**Phase 1完成后**：
- 用户在YunPat目录运行`deepseek`，启动的已经是"专利版"TUI
- 可以通过自然语言调用`mcp_yunpat_patent_search`、`mcp_yunpat_patent_draft_full`等工具
- 专利Skills自动加载，注入system prompt

### Phase 2：深度融合（3-9个月）

**目标**：让YunPat的专业Agent逻辑真正跑在DeepSeek-TUI的执行引擎里，而非通过网络边界调用。

| 任务 | 工作量 | 技术方案 |
|------|--------|----------|
| **编排器融合** | 2-3个月 | YunPat的`OrchestratorAgent`通过`orchestrator-adapter`（Rust网关）接入`deepseek-core` |
| **工具层统一** | 1-2个月 | YunPat的`patent-tools`、`document-tools`注册到DeepSeek-TUI的类型化工具注册表 |
| **记忆层对齐** | 1个月 | 统一检查点格式，让YunPat的HITL检查点与DeepSeek-TUI的session恢复互通 |
| **RLM并行加速** | 2-3周 | 接入DeepSeek-TUI的RLM引擎，批量专利分析速度数量级提升 |

**Phase 2完成后**：
- 专利任务的意图识别、任务规划、异常处理由YunPat Orchestrator负责
- 执行、session管理、审批门控由DeepSeek-TUI负责
- 两者的记忆系统互通，"写到一半关掉终端，下次继续"

### Phase 3：宪法规划层（9-18个月）

**目标**：为知识产权场景实现专业化宪法约束，保障专利文件安全与合规。

| 任务 | 工作量 | 技术方案 |
|------|--------|----------|
| **IP宪法定义** | 1个月 | 定义专利场景宪法原则（不泄露、权利要求保守、引用规范等） |
| **Critique-Revision循环** | 2-3个月 | 复用DeepSeek-TUI的pre-turn routing call，在每次输出前做宪法合规检查 |
| **宪法门控** | 2个月 | 扩展`deepseek-execpolicy`，在YOLO模式下也保持专利数据约束 |
| **数据主权强制** | 1-2个月 | 当工具调用涉及专利敏感数据时，强制路由到本地Ollama或私有部署 |
| **审计层** | 2个月 | 完整的操作审计日志，满足法律合规要求 |

**IP宪法示例条款**：
- 技术交底内容不得出现在API请求的prompt里（本地优先原则）
- 权利要求措辞只能收紧，不能主动拓宽（防止无效风险）
- 引用对比文件时必须附带原始检索结果（可溯源原则）
- 涉及竞争对手专利的分析不得直接复制权利要求文字

**Phase 3完成后**：
- 每个任务执行前都经过宪法合规检查
- 违规操作被自动阻断，即使YOLO模式也无法绕过
- 完整的审计日志，可追溯每一次操作

### Phase 4：产品化平台（18-36个月）

**目标**：成为面向专利代理所/律所/企业IP部门的行业级AI智能体平台。

| 任务 | 工作量 | 商业价值 |
|------|--------|----------|
| **企业部署** | 3-6个月 | 多租户隔离、私有化部署包、PostgreSQL企业级迁移 |
| **并行加速** | 1-2个月 | RLM引擎批量分析（同时检索CN/US/EP三个数据库） |
| **知识飞轮** | 持续 | 用户成功案例沉淀为新Skills，越用越强 |
| **商业化** | 3-6个月 | SaaS订阅、私有部署授权、API服务 |
| **扩展领域** | 6-12个月 | 商标、著作权、合同、数据合规、国际专利（PCT/USPTO/EPO） |

**Phase 4完成后**：
- 面向专利代理所/律所/企业IP部门的完整产品
- 从"工具"升级为"平台"
- 构建中国法律垂直领域的Constitutional AI OS

### 4.2 推荐策略：分阶段演进

#### 第一阶段（立即执行）：保持独立，建立桥梁

**不改变现有仓库结构**：
- DeepSeek-TUI继续作为独立开源项目维护（保持与上游Hmbown的同步）
- YunPat继续作为私有仓库维护

**建立连接**：
- 在YunPat仓库中添加`.deepseek/`目录（配置、命令、Skills）
- 在DeepSeek-TUI中添加YunPat作为MCP server的文档和示例配置

#### 第二阶段（Phase 2完成后）：创建融合仓库

**新建仓库**：`github.com/xujian519/YunPat-OS`（或`Patent-Agent-OS`）

**仓库结构**：
```
yunpat-os/
├── .github/                 # CI/CD工作流
├── docs/                    # 融合项目文档
├── crates/                  # Rust核心（从DeepSeek-TUI演进）
│   ├── cli/                 # CLI入口（yunpat命令）
│   ├── tui/                 # TUI界面（ratatui）
│   ├── core/                # 核心运行时
│   ├── constitutional/      # 宪法规划层（新增）
│   └── ...
├── packages/                # TypeScript业务层（从YunPat演进）
│   ├── agents/              # 专业Agent（29个）
│   ├── orchestrator/        # 编排器
│   ├── patent-database/     # 专利数据库
│   └── ...
├── knowledge-base/          # 知识库（4382个文件）
├── constitutional/          # 宪法规则文件
│   ├── patent-law.yaml      # 专利法规则
│   ├── compliance-rules.yaml # 合规规则
│   └── audit-policies.yaml  # 审计策略
├── docker/                  # Docker部署
└── README.md
```

**为什么新建仓库而不是合并到现有仓库？**
1. **技术栈异构**：Rust + TypeScript双语言，各自的构建工具链独立
2. **代码量巨大**：两个项目合计~100,000行代码，强行合并会导致仓库爆炸
3. **开源策略不同**：DeepSeek-TUI保持开源，YunPat核心可能保持私有
4. **版本节奏不同**：DeepSeek-TUI需要频繁同步上游，YunPat版本节奏独立

#### 第三阶段（Phase 4完成后）：品牌升级

**产品品牌**：`YunPat Constitutional OS`（云铂宪法智能体操作系统）

**仓库策略**：
- **开源部分**（github.com/yunpat-os/core）：TUI、MCP、宪法引擎框架
- **私有部分**（企业内部）：专利数据库、知识库、专业Agent
- **商业部分**（SaaS）：云端API服务

**URL策略**：
- 注册域名：`yunpat.ai` 或 `patent-agent.ai`
- 文档站点：`docs.yunpat.ai`
- API文档：`api.yunpat.ai`

### 4.3 为什么不把YunPat合并到DeepSeek-TUI仓库？

| 风险点 | 说明 |
|--------|------|
| **上游同步困难** | DeepSeek-TUI是fork项目，需要持续同步Hmbown的上游更新。混入TypeScript代码会让同步变得极其困难 |
| **构建系统冲突** | Cargo和pnpm是完全不同的构建体系，强行合并会导致CI/CD复杂度指数级增长 |
| **开发者体验** | Rust开发者和TypeScript开发者的工具链完全不同，合并后贡献门槛提高 |
| **代码审查** | 跨语言代码审查需要两类专家，review效率降低 |
| **发布节奏** | Rust crate发布到crates.io，TypeScript包发布到npm，节奏不同 |

### 4.4 为什么不直接把DeepSeek-TUI改名为YunPat？

| 风险点 | 说明 |
|--------|------|
| **社区资产** | DeepSeek-TUI作为开源项目已有一定的社区认知度，改名会损失这部分资产 |
| **上游依赖** | 持续从Hmbown/DeepSeek-TUI同步更新，改名后同步逻辑复杂 |
| **通用性** | DeepSeek-TUI的通用编码Agent能力不应被专利业务绑架，保持独立有利于两个方向各自发展 |

### 5.2 关键接合点设计

**接合点A：MCP协议（Phase 1）**
- YunPat的`packages/mcp-server`通过stdio暴露专利工具
- DeepSeek-TUI通过`~/.deepseek/mcp.json`注册YunPat MCP server
- 工具内部命名：`mcp_yunpat_patent_search`、`mcp_yunpat_patent_draft_full`

**接合点B：Hooks机制（Phase 2）**
- DeepSeek-TUI的`message_submit` Hook触发YunPat的`IntentRecognizer`
- DeepSeek-TUI的`on_error` Hook触发YunPat的`ExceptionHandler`
- 通过JSONL格式交换指令（set_mode、prepend_context、load_skill）

**接合点C：agent_spawn（Phase 2）**
- 复杂专利任务通过`agent_spawn`生成子Agent
- 子Agent开启新的干净session，处理独立任务
- 结果回写主session，保持主session上下文精简

**接合点D：宪法门控（Phase 3）**
- 在`deepseek-execpolicy`中扩展宪法约束
- 即使YOLO模式，专利敏感数据操作也需要审批
- 通过`deepseek-hooks`的`post_response_hook`做输出合规扫描

### 5.3 推荐语言分工

| 层级 | 语言 | 原因 |
|------|------|------|
| **TUI/CLI** | Rust | 高性能终端渲染，内存安全 |
| **宪法引擎** | Rust | 高性能规则匹配，安全关键 |
| **编排器** | TypeScript | 业务逻辑复杂，快速迭代 |
| **专业Agent** | TypeScript | 已有29个Agent，6万行代码 |
| **数据库** | PostgreSQL | 专利数据存储 |
| **知识图谱** | Neo4j + Python | 图算法、ML模型 |
| **Embedding** | Python | 模型推理 |

## 七、风险与缓解措施

## 九、总结

### 核心判断

1. **融合可行**：两个项目天然互补，YunPat提供专利大脑，DeepSeek-TUI提供执行躯体
2. **不要合并代码**：技术栈异构，强行合并会导致复杂度爆炸。通过MCP/gRPC/Hooks建立跨语言边界
3. **新建融合仓库**：Phase 2完成后创建`YunPat-OS`仓库，Rust核心层+TypeScript业务层+知识库
4. **宪法规划是核心壁垒**：Phase 3的宪法引擎是区别于其他专利AI工具的关键差异化
5. **商业化路径清晰**：从SaaS到私有部署到API服务，目标客户从专利代理人扩展到企业IP部门
