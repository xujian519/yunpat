## 仓库现状梳理

### 架构概览

该项目是一个 13-crate 的 Rust workspace，依赖分层如下：

- **Layer 0（叶节点）**：`deepseek-protocol`、`deepseek-config`、`deepseek-state`、`deepseek-tui-core`
- **Layer 1**：`deepseek-tools`、`deepseek-mcp`、`deepseek-hooks`、`deepseek-execpolicy`
- **Layer 2**：`deepseek-agent`
- **Layer 3**：`deepseek-core`（Agent 主循环）
- **Layer 4**：`deepseek-app-server`、`deepseek-tui`
- **Layer 5**：`deepseek-tui-cli`（入口 `deepseek` 命令）

### 核心执行策略

三种运行模式：Plan（只读调查，不做变更）、Agent（多步工具调用 + 审批门控）、YOLO（自动审批全部工具，解除工作区边界限制）。

`deepseek-execpolicy` crate 负责执行策略，依赖 `deepseek-protocol`；`deepseek-hooks` 也依赖 `deepseek-protocol`，是插入行为约束的关键切面。

------

## 什么是"宪法规划"（Constitutional Planning）？

在 AI Agent 领域，**宪法规划**（Constitutional AI / Constitutional Planning）指的是在 Agent 行动之前或执行中，通过一组预定义的"宪法原则"（principles）对 Agent 的行为进行约束和自我审查，典型流程是：

1. Agent 生成一个行动计划或回复草稿
2. 对照"宪法"规则进行自我批评（critique）
3. 修订草稿，使其符合原则
4. 输出修订后的安全/合规版本

------

## 当前代码与宪法规划的差距分析

### 已有基础（有利条件）

| 现有能力                        | 与宪法规划的关联                                             |
| ------------------------------- | ------------------------------------------------------------ |
| `deepseek-execpolicy` crate     | 已有"执行策略"层，是挂载宪法约束的天然位置                   |
| `deepseek-hooks` crate          | Hook 机制可插入 pre/post 行为检查                            |
| Plan 模式                       | 已有"先计划、再执行"的结构雏形                               |
| `deepseek-core` 中的 Agent 循环 | 多轮对话状态管理，可插入 critique 轮次                       |
| Thinking mode streaming         | 支持 `reasoning_content` 流式输出，可用于宪法自我批评过程的可视化 |

### 缺失的核心组件（下一步需要做什么）

**1. 宪法文件加载机制（Constitution Loader）**

目前 Skills 系统（`SKILL.md`）已支持加载自定义指令，但没有专门的"宪法"概念。需要在 `deepseek-config` 中新增：

- 支持从 `~/.deepseek/constitution.md` 或项目级 `.deepseek/constitution.md` 加载原则列表
- 区分"全局宪法"（安全红线）和"项目宪法"（风格/规范约束）

**2. Critique-Revision 循环（自我批评轮次）**

这是宪法规划的核心。需要在 `deepseek-core` 的 Agent 循环中增加：

```
用户请求
    ↓
草稿生成（draft turn）
    ↓
宪法 critique（对照原则，调用一次轻量 LLM 检查）
    ↓
如果违规 → revision turn（重新生成）
    ↓
最终输出
```

这个结构与现有的 `auto` 模式 routing call 非常相似——在真实 turn 发出之前，先发出一个小型的 `deepseek-v4-flash` routing call 来选择模型和思考级别——可以复用这套 pre-turn call 的基础设施，在 routing call 之后、主请求之前插入 critique 调用。

**3. `deepseek-execpolicy` 中的宪法门控（Constitutional Gate）**

`deepseek-execpolicy` 目前依赖 `deepseek-protocol`，处理工具调用的审批逻辑。需要扩展：

- 对**工具调用结果**进行宪法合规性检查，不仅仅是"是否允许执行"，还要检查"执行意图是否符合原则"
- 对**危险操作**（文件删除、shell 命令、git force push 等）在 YOLO 模式下也保持宪法约束

**4. `deepseek-hooks` 中的 pre/post 宪法 Hook**

`deepseek-hooks` 目前的 Hook 架构可以扩展为：

- `pre_turn_hook`：在发出每个 turn 前，将宪法原则注入 system prompt
- `post_response_hook`：对模型输出做宪法合规扫描
- `on_tool_call_hook`：在工具调用前进行意图对齐检查

**5. TUI 可视化层**

目前 TUI 会显示所选的 routing 路由，宪法规划也需要类似的可观测性：

- 在 transcript 中显示 critique 步骤（类似 thinking block 的视觉区分）
- 对被宪法拒绝或修改的 action 做高亮提示

**6. 测试与评估体系**

宪法规划需要专门的评估机制：

- 构建"红队测试"用例，验证宪法约束是否真正有效
- 针对 `deepseek-execpolicy` 的单元测试，覆盖宪法违规场景

------

## 推荐实施路径（按优先级）

**第一步（最小可行实现）**：在 `deepseek-config` 中加入宪法文件解析，并在 `deepseek-core` 的 Agent 循环里，将宪法原则作为额外的 system prompt 片段注入每个 turn。这是最低成本的切入点，复用现有 Skills 的文件加载逻辑。

**第二步**：在 `deepseek-hooks` 中实现 `post_response_hook`，用 `deepseek-v4-flash` 对每次模型输出做轻量 critique pass，复用现有的 routing call 基础设施。

**第三步**：扩展 `deepseek-execpolicy`，在工具调用审批逻辑中加入宪法门控层，确保即使在 YOLO 模式下也有宪法兜底。

**第四步**：完善 TUI 的宪法流程可视化，并构建评估测试集。

总结一句话：**该仓库的 `execpolicy` + `hooks` + `core` 三层架构为宪法规划提供了相当好的骨架，核心缺口是宪法文件加载、critique-revision 轮次逻辑，以及将宪法约束与现有 pre-turn routing call 机制深度整合。**

## 融合分析：两个项目的本质互补

先说清楚为什么这两个项目天然适合深度融合，而不只是"用 DeepSeek-TUI 作为命令行工具"那种浅层集成：

**YunPat 有什么，缺什么？** YunPat 已经是一个高度成熟的专利领域 AI 系统（75% 完成度，62000 行核心代码，29 个专业 Agent），拥有世界级的领域知识积累（4382 个专利知识库文件）和完整的五层架构。但它在 **执行层** 上相对薄弱——TUI 才 80%，CLI 会被 TUI 替代，整个终端体验是补课状态。

**DeepSeek-TUI 有什么，缺什么？** DeepSeek-TUI 是一个执行层极其成熟的通用编程 Agent（完整的 Rust 工具链、沙箱安全、MCP 双向、RLM 并行、session 管理），但它是 **通用的**——没有任何领域知识，面对专利撰写、审查答复这类专业任务完全不知所措。

融合的本质是：**YunPat 的专利大脑 + DeepSeek-TUI 的执行躯体**。

## Phase 1：接口对齐（0–3 个月）

这一阶段的核心原则是**最小侵入，快速打通**，不重构任何东西，只建立三条桥梁。

**MCP 对接**是最自然的切入点。YunPat 已有 `packages/mcp-server`（6 个源文件），DeepSeek-TUI 既是 MCP 客户端又是 MCP 服务端。做法很简单：把 YunPat 的 MCP server 启动后，在 `~/.deepseek/mcp.json` 里注册，DeepSeek-TUI 就能直接调用 `patent-writer`、`patent-analyzer`、`patent-responder` 等所有专利 Agent 作为工具。这个工作量大约 1–2 周。

**Skills 格式统一**：两个项目都有 Skills 系统，但格式不同。YunPat 是 TypeScript 的 `packages/skills`（20 个源文件），DeepSeek-TUI 是目录 + `SKILL.md` 的 Markdown 格式。需要写一个转换器，把 YunPat 的 Skills 导出为 DeepSeek-TUI 兼容的 `SKILL.md` 格式，或者在 YunPat 的 Skills 系统里直接支持输出 `SKILL.md`。

**TUI 层取舍**：这是一个需要果断决策的地方。YunPat 的 TUI 是 TypeScript + Ink（React 风格，17 个源文件，80% 完成），DeepSeek-TUI 的 TUI 是 Rust + ratatui（生产就绪）。建议直接放弃 YunPat 的 Ink TUI，将精力集中在用 Rust TUI 做专利场景的定制，这能省掉大量重复工作。

------

## Phase 2：深度融合（3–9 个月）

这一阶段的目标是让 YunPat 的专业 Agent 逻辑真正跑在 DeepSeek-TUI 的执行引擎里，而不是通过 MCP 的网络边界来调用。

**编排器融合**是核心难点，也是最有价值的工作。YunPat 的 `orchestrator` 包（intent 识别 → task planning → routing → exception 处理）需要与 `deepseek-core` 的 Agent 主循环整合。可行方案是：将 YunPat 的 `OrchestratorAgent` 编译为 WASM 或通过 YunPat 已有的 `orchestrator-adapter`（Rust 网关桥接，4 个源文件）直接接入 `deepseek-core`。这个 adapter 的存在表明 YunPat 的架构师已经预见到了这种跨语言集成需求。

**工具层统一**：把 YunPat 的 `patent-tools`（9 个文件）、`document-tools`（14 个文件）、`image-tools`（3 个文件）注册到 DeepSeek-TUI 的类型化工具注册表里。这需要为每个工具写 Rust 侧的 FFI 绑定或 gRPC 适配——YunPat 已有 `packages/grpc-server`（4 个文件）和 `packages/rust-tools`，基础设施已经存在。

**记忆层对齐**：YunPat 的 HITL 检查点、时间旅行（Memory 层）与 DeepSeek-TUI 的 `deepseek --continue` session 恢复机制在语义上高度重叠。需要设计一个统一的检查点格式，让两个系统都能读写，实现"专利撰写到一半，关掉终端，下次继续"这样的用户体验。

------

## Phase 3：宪法规划层（9–18 个月）

这是本次讨论的重点，也是这个融合项目能建立真正壁垒的地方。

知识产权场景对宪法规划的需求比通用编程 Agent 强烈得多。原因是：专利文件一旦泄露或写错，可能造成数十万乃至数亿的损失；审查答复一旦引用了错误的先有技术，可能导致专利无效。通用 Agent 的宪法关注"不输出有害内容"，专利 Agent 的宪法关注"不泄露技术交底、不降低权利要求保护范围、引用先有技术必须准确"。

**IP 宪法的具体条款示例**：

- 技术交底内容不得出现在 API 请求的 prompt 里（本地优先原则的技术实现）
- 权利要求措辞只能收紧，不能主动拓宽（防止无效风险）
- 引用对比文件时必须附带原始检索结果（可溯源原则）
- 涉及竞争对手专利的分析不得直接复制权利要求文字

**Critique-Revision 循环实现**：复用 DeepSeek-TUI 现有的 pre-turn routing call 基础设施——在主请求之前，先用 `deepseek-v4-flash` 对草稿做一次宪法合规检查，返回违规列表，如有违规则触发修订循环。这与 auto 模式的路由调用在代码路径上非常相似，改造成本低。

**数据主权约束**：YunPat 的"本地优先承诺"（100% 可本地部署）需要在执行层有强制保证，不能只靠文档声明。在 `deepseek-execpolicy` 里加入一条规则：当工具调用涉及专利敏感数据时，强制路由到本地 Ollama 或私有部署的 DeepSeek，而不是外部 API——这是宪法门控的具体实现。

------

## Phase 4：产品化平台（18–36 个月）

到这个阶段，融合后的系统应该是一个真正面向知识产权行业的 AI Agent 平台，而不只是一个开发者工具。

**DeepSeek-TUI 的 RLM 并行引擎**（最多 16 个并行 `deepseek-v4-flash` 子任务）对专利场景有巨大价值——一件专利需要同时检索 CN/US/EP 三个数据库，可以完全并行。YunPat 目前的 `PatentSearchAgentV3` 是串行的，接入 RLM 后批量分析速度可以有数量级的提升。

**知识飞轮**是长期壁垒所在。YunPat 已有 4382 个专利知识库文件，用户每次使用产生的成功案例（哪类技术交底对应哪种权利要求写法、哪类审查意见用什么答复策略最有效）都可以沉淀为新的 Skills，形成越用越强的飞轮效应。

------

## 最需要注意的融合风险

有一点值得特别提醒：**两个项目的技术栈是异构的**——YunPat 是 TypeScript（70%）+ Rust（30%），DeepSeek-TUI 是纯 Rust（99.3%）。这意味着融合不是"合并代码"，而是"建立稳定的跨语言边界"。YunPat 现有的 `orchestrator-adapter`（Rust 网关）和 `grpc-server` 是正确的方向，应该把这两个包作为融合的主要接口层来重点建设，而不是试图把 TypeScript 业务逻辑重写成 Rust。

## MCP 对接全步骤

整个对接分三层，从外到内依次是：**配置层**（让 DeepSeek-TUI 发现 YunPat）、**协议层**（让 YunPat 正确实现 MCP stdio）、**工具层**（让专利 Agent 暴露为可调用工具）。

------

### 第一步：确认 YunPat mcp-server 的现状

YunPat 的 `packages/mcp-server` 已有 6 个源文件，需要先弄清楚它目前实现的是哪种传输模式。DeepSeek-TUI 支持两种 MCP 传输方式：本地 stdio 进程模式（`"command": "node", "args": [...]`）和 HTTP URL 模式（`"url": "http://localhost:3000/mcp"`）。

检查命令：

```bash
# 在 yunpat 目录下
cat packages/mcp-server/src/index.ts
# 确认是否有 StdioServerTransport 或 SSEServerTransport
```

如果已有 `StdioServerTransport`，可以直接走 stdio 模式，配置最简单、延迟最低。如果是 HTTP/SSE，走 URL 模式注册即可，但需要保持服务器常驻。**优先推荐 stdio 模式**，原因是：无需单独启动服务进程，DeepSeek-TUI 会自动管理其生命周期。

------

### 第二步：确保 YunPat mcp-server 满足 MCP stdio 规范

YunPat 的 `packages/mcp-server` 需要能以 stdio 方式运行并正确暴露所有专利工具。标准实现模板如下（基于 `@modelcontextprotocol/sdk`）：

```typescript
// packages/mcp-server/src/index.ts
import { Server } from '@modelcontextprotocol/sdk/server/index.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'
import {
  ListToolsRequestSchema,
  CallToolRequestSchema,
} from '@modelcontextprotocol/sdk/types.js'

// 导入 YunPat 各专利 Agent
import { PatentWriterAgent }    from '@yunpat/patent-writer'
import { PatentSearchAgent }    from '@yunpat/search'
import { PatentAnalyzerAgent }  from '@yunpat/patent-analyzer'
import { PatentResponderAgent } from '@yunpat/patent-responder'

const server = new Server(
  { name: 'yunpat', version: '0.1.0' },
  { capabilities: { tools: {} } }
)

// ① 工具清单：DeepSeek-TUI 启动时会 list 一次，用于填充工具注册表
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: 'patent_draft_full',
      description: '完整专利撰写工作流：发明理解→检索→说明书→权利要求→摘要',
      inputSchema: {
        type: 'object',
        properties: {
          title:       { type: 'string', description: '发明名称' },
          field:       { type: 'string', description: '技术领域' },
          disclosure:  { type: 'string', description: '技术交底书内容（Markdown）' },
        },
        required: ['title', 'field', 'disclosure'],
      },
    },
    {
      name: 'patent_search',
      description: '专利检索，覆盖7500万CN专利+全球专利库，返回检索报告',
      inputSchema: {
        type: 'object',
        properties: {
          title:      { type: 'string' },
          field:      { type: 'string' },
          disclosure: { type: 'string' },
        },
        required: ['title', 'field'],
      },
    },
    {
      name: 'patent_analyze',
      description: '专利对比分析，自动检索对比文件，返回新颖性/创造性评估',
      inputSchema: {
        type: 'object',
        properties: {
          patent_text: { type: 'string', description: '待分析专利全文' },
          claim_focus: { type: 'string', description: '重点分析的权利要求编号，可选' },
        },
        required: ['patent_text'],
      },
    },
    {
      name: 'patent_respond',
      description: '审查意见答复，自动检索先例案例，生成答复策略和修改建议',
      inputSchema: {
        type: 'object',
        properties: {
          office_action: { type: 'string', description: '审查意见通知书全文' },
          patent_claims: { type: 'string', description: '当前权利要求书' },
        },
        required: ['office_action', 'patent_claims'],
      },
    },
  ],
}))

// ② 工具执行：DeepSeek-TUI 决定调用时触发
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params

  try {
    switch (name) {
      case 'patent_draft_full': {
        const agent = new PatentWriterAgent({ /* 从环境变量读取 LLM 配置 */ })
        const result = await agent.execute(args)
        return { content: [{ type: 'text', text: JSON.stringify(result, null, 2) }] }
      }
      case 'patent_search': {
        const agent = new PatentSearchAgent({ /* ... */ })
        const result = await agent.execute(args)
        return { content: [{ type: 'text', text: JSON.stringify(result, null, 2) }] }
      }
      case 'patent_analyze': {
        const agent = new PatentAnalyzerAgent({ /* ... */ })
        const result = await agent.execute(args)
        return { content: [{ type: 'text', text: JSON.stringify(result, null, 2) }] }
      }
      case 'patent_respond': {
        const agent = new PatentResponderAgent({ /* ... */ })
        const result = await agent.execute(args)
        return { content: [{ type: 'text', text: JSON.stringify(result, null, 2) }] }
      }
      default:
        throw new Error(`未知工具: ${name}`)
    }
  } catch (err) {
    // MCP 规范：工具执行错误通过 content 返回，不抛异常
    return {
      content: [{ type: 'text', text: `执行失败: ${(err as Error).message}` }],
      isError: true,
    }
  }
})

// ③ stdio 传输层启动
async function main() {
  const transport = new StdioServerTransport()
  await server.connect(transport)
  // 注意：不要向 stdout 输出任何调试信息！stdio 模式下 stdout 是 MCP 协议通道
  // 调试信息必须写到 stderr 或日志文件
  process.stderr.write('YunPat MCP server started\n')
}

main().catch((e) => {
  process.stderr.write(`Fatal: ${e}\n`)
  process.exit(1)
})
```

在 `package.json` 里加一个专门的 MCP 启动脚本：

```json
{
  "scripts": {
    "mcp": "tsx packages/mcp-server/src/index.ts",
    "mcp:build": "tsc -p packages/mcp-server/tsconfig.json",
    "mcp:start": "node packages/mcp-server/dist/index.js"
  }
}
```

### 第三步：在 DeepSeek-TUI 侧注册 YunPat

MCP 配置文件路径为 `~/.deepseek/mcp.json`，可以用 `deepseek-tui mcp init` 生成初始文件。

```bash
# 先确认 mcp.json 路径
deepseek doctor
# 输出中会有：MCP config: /Users/yourname/.deepseek/mcp.json

# 初始化（如果还没有）
deepseek-tui mcp init
```

然后编辑 `~/.deepseek/mcp.json`：

```json
{
  "timeouts": {
    "connect_timeout": 15,
    "execute_timeout": 300,
    "read_timeout": 600
  },
  "servers": {
    "yunpat": {
      "command": "node",
      "args": ["/absolute/path/to/yunpat/packages/mcp-server/dist/index.js"],
      "env": {
        "DEEPSEEK_API_KEY": "${DEEPSEEK_API_KEY}",
        "NODE_ENV": "production"
      },
      "disabled": false
    }
  }
}
```

几个关键参数的说明：

`execute_timeout` 设为 300 秒是因为专利撰写工作流（`patent_draft_full`）涉及多步 Agent 调用，可能需要 3–5 分钟。默认的 execute_timeout 只有 60 秒，对专利场景远远不够，必须显式调大。

路径必须用绝对路径，因为 DeepSeek-TUI 启动 MCP 子进程时的工作目录是不确定的。

**如果不想每次改 JSON，也可以用命令行注册：**

```bash
deepseek-tui mcp add yunpat \
  --command "node" \
  --arg "/absolute/path/to/yunpat/packages/mcp-server/dist/index.js"
```

------

## 两个 Skills 系统的核心差异

先把问题说清楚，才能设计出正确的解法。

**DeepSeek-TUI 的 Skills 是什么：** 每个 Skill 是一个目录，目录里放一个 `SKILL.md` 文件，文件顶部是 YAML frontmatter（`name` + `description`），下面是给 Agent 的自然语言指令。Agent 在任务描述匹配时通过 `load_skill` 工具自动加载对应的 `SKILL.md` 内容注入上下文。 本质上是**运行时上下文注入**——把一段指令文本追加进 system prompt。

**YunPat 的 Skills 是什么：** 根据 README，`packages/skills`（20 个源文件）是 TypeScript 代码包，Skills 在 YunPat 架构里承担的是**可组合的工作流片段**角色，每个 Skill 可能包含工具调用逻辑、参数验证、前后处理钩子等程序化行为，远比一个 Markdown 文件复杂。

这就是核心矛盾：**DeepSeek-TUI 的 Skill = 一段提示词文本，YunPat 的 Skill = 一段可执行代码。** 直接互转是不可能的，需要一个分层策略。

------

## 统一策略：分三层处理

不同类型的 YunPat Skill 走不同的转换路径：

```
YunPat Skills
    │
    ├── 纯指令类（工作流步骤、撰写规范、质量标准）
    │       └──→  直接导出为 SKILL.md  ✅ 完全兼容
    │
    ├── 工具调用类（需要调用 patent_search、patent_analyze 等）
    │       └──→  SKILL.md + MCP 工具引用  ✅ 结合上一步的 MCP 对接
    │
    └── 程序逻辑类（含状态、循环、条件分支的复杂 Skill）
            └──→  保留在 YunPat 侧，通过 MCP 工具暴露  ✅ 不强行转换
```

------

## 第一步：建立统一的 Skill 源格式

在 YunPat 的 `packages/skills` 里新增一个**规范化 frontmatter 约定**，让每个 Skill 源文件都携带足够的元数据，使得导出 `SKILL.md` 成为无损操作。

在 `packages/skills/src/types.ts` 里定义统一接口：

```typescript
// packages/skills/src/types.ts

export interface SkillFrontmatter {
  /** Skill 唯一标识，用于目录名和 /skill <name> 命令 */
  name: string
  /** 触发描述：DeepSeek 用这段文字判断是否自动 load_skill */
  description: string
  /** 版本，用于增量同步判断 */
  version: string
  /** 分类标签 */
  tags?: string[]
  /**
   * 导出策略：
   * - 'markdown'  → 直接导出为 SKILL.md（纯指令类）
   * - 'mcp-ref'   → 导出含 MCP 工具引用的 SKILL.md（工具调用类）
   * - 'mcp-only'  → 不导出 SKILL.md，仅通过 MCP 工具暴露（程序逻辑类）
   */
  export: 'markdown' | 'mcp-ref' | 'mcp-only'
  /** export='mcp-ref' 时声明依赖的 MCP 工具列表 */
  mcpTools?: string[]
}

export interface YunPatSkill {
  frontmatter: SkillFrontmatter
  /** 给 Agent 看的 Markdown 正文（指令内容） */
  content: string
  /** 可选：程序化执行逻辑（仅 YunPat 运行时使用） */
  execute?: (input: unknown) => Promise<unknown>
}
```

## 第二步：编写导出器

在 `packages/skills/src/exporter.ts` 里实现从 YunPat Skill 到 `SKILL.md` 的导出逻辑：

```typescript
// packages/skills/src/exporter.ts
import fs from 'node:fs/promises'
import path from 'node:path'
import type { YunPatSkill } from './types.js'

/**
 * 将 YunPat Skills 导出为 DeepSeek-TUI 兼容的 SKILL.md 目录结构
 *
 * 输出结构：
 *   <outputDir>/
 *     patent-writer/
 *       SKILL.md
 *     patent-responder/
 *       SKILL.md
 *     ...
 */
export async function exportSkills(
  skills: YunPatSkill[],
  outputDir: string,
): Promise<{ exported: string[]; skipped: string[] }> {
  const exported: string[] = []
  const skipped: string[] = []

  await fs.mkdir(outputDir, { recursive: true })

  for (const skill of skills) {
    const { frontmatter, content } = skill

    if (frontmatter.export === 'mcp-only') {
      skipped.push(frontmatter.name)
      continue
    }

    const skillDir = path.join(outputDir, frontmatter.name)
    await fs.mkdir(skillDir, { recursive: true })

    let body = content

    // mcp-ref 类型：在正文末尾追加 MCP 工具说明段
    if (frontmatter.export === 'mcp-ref' && frontmatter.mcpTools?.length) {
      body += '\n\n## 可用工具\n\n'
      body += '本 Skill 依赖以下 MCP 工具，在需要时直接调用：\n\n'
      for (const tool of frontmatter.mcpTools) {
        body += `- \`${tool}\`\n`
      }
    }

    // 拼接标准 SKILL.md 格式
    const skillMd = [
      '---',
      `name: ${frontmatter.name}`,
      `description: ${frontmatter.description}`,
      frontmatter.tags?.length ? `tags: [${frontmatter.tags.join(', ')}]` : null,
      `version: ${frontmatter.version}`,
      '---',
      '',
      body,
    ]
      .filter((line) => line !== null)
      .join('\n')

    await fs.writeFile(path.join(skillDir, 'SKILL.md'), skillMd, 'utf-8')
    exported.push(frontmatter.name)
  }

  return { exported, skipped }
}

/**
 * 增量同步：只导出有版本变化的 Skill
 * 通过读取现有 SKILL.md 的 version 字段来判断
 */
export async function syncSkills(
  skills: YunPatSkill[],
  outputDir: string,
): Promise<{ updated: string[]; unchanged: string[]; skipped: string[] }> {
  const updated: string[] = []
  const unchanged: string[] = []
  const skipped: string[] = []

  for (const skill of skills) {
    if (skill.frontmatter.export === 'mcp-only') {
      skipped.push(skill.frontmatter.name)
      continue
    }

    const skillMdPath = path.join(outputDir, skill.frontmatter.name, 'SKILL.md')

    try {
      const existing = await fs.readFile(skillMdPath, 'utf-8')
      const versionMatch = existing.match(/^version:\s*(.+)$/m)
      const existingVersion = versionMatch?.[1]?.trim()

      if (existingVersion === skill.frontmatter.version) {
        unchanged.push(skill.frontmatter.name)
        continue
      }
    } catch {
      // 文件不存在，需要新建
    }

    // 版本不同或文件不存在，重新导出
    await exportSkills([skill], outputDir)
    updated.push(skill.frontmatter.name)
  }

  return { updated, unchanged, skipped }
}
```

## 第四步：构建 CLI 同步命令

在 `packages/cli/src/commands/skills-sync.ts` 里加一个同步命令，让开发者和 CI 都能一键同步：

```typescript
// packages/cli/src/commands/skills-sync.ts
import { Command } from 'commander'
import os from 'node:os'
import path from 'node:path'
import { syncSkills, exportSkills } from '@yunpat/skills/exporter'
import { allSkills } from '@yunpat/skills'  // 导出所有 Skill 的注册表

export const skillsSyncCommand = new Command('skills:sync')
  .description('将 YunPat Skills 同步到 DeepSeek-TUI 格式')
  .option(
    '--output <dir>',
    '目标目录',
    // 默认写到 DeepSeek-TUI 能自动发现的路径之一
    path.join(process.cwd(), '.deepseek', 'skills'),
  )
  .option('--global', '同步到全局 ~/.deepseek/skills（对所有项目生效）')
  .option('--force', '强制全量导出，忽略版本缓存')
  .action(async (opts) => {
    const outputDir = opts.global
      ? path.join(os.homedir(), '.deepseek', 'skills')
      : opts.output

    console.log(`同步目标：${outputDir}`)

    if (opts.force) {
      const { exported, skipped } = await exportSkills(allSkills, outputDir)
      console.log(`✅ 全量导出：${exported.join(', ')}`)
      if (skipped.length) console.log(`⏭️  跳过（mcp-only）：${skipped.join(', ')}`)
    } else {
      const { updated, unchanged, skipped } = await syncSkills(allSkills, outputDir)
      if (updated.length)   console.log(`🔄 已更新：${updated.join(', ')}`)
      if (unchanged.length) console.log(`✔️  无变化：${unchanged.join(', ')}`)
      if (skipped.length)   console.log(`⏭️  跳过：${skipped.join(', ')}`)
    }
  })
```

------

## 先把问题说清楚：两个 TUI 的本质差异

**DeepSeek-TUI 的 ratatui TUI**：核心交互应用，包含 ratatui 渲染循环、Engine 状态机和工具注册表。 整体架构是 `dispatcher → ratatui interface ↔ async engine ↔ streaming client`，工具调用经过类型化注册表路由，结果流式回写 transcript。 这是一个**生产就绪的 Rust 原生 TUI**，已有 session 管理、workspace 回滚、LSP 诊断、cost tracking、多语言本地化等完整功能。

**YunPat 的 Ink TUI**：`packages/tui`（17 个源文件，80% 完成），React/Ink 风格，TypeScript 实现。当前功能不完整，README 明确标注"持续完善中"，且 CLI 工具将被 TUI 替代——说明 YunPat 自己也认为 Ink TUI 是过渡方案。

**真实问题不是"如何合并两个 TUI"，而是"是否值得合并"**。答案是：**不值得合并，应该直接放弃 Ink TUI，以 DeepSeek-TUI 的 ratatui TUI 为主体，在上面做专利场景定制扩展。**

原因有三：Ink TUI 才 80%，没有 session 恢复、workspace 回滚、LSP 等核心能力；两者技术栈（TypeScript vs Rust）完全异构，真正的"合并"意味着重写其中一个；DeepSeek-TUI 的 ratatui TUI 是 Claude Code 级别的终端体验，YunPat 的目标用户本就需要这种质量。

**正确的统一策略是：用 DeepSeek-TUI ratatui TUI 作为唯一界面，通过三个切入点注入专利场景能力。**

------

## 三个切入点先用一张结构图把整体方案说清楚。三个切入点的实施顺序和工作量差异很大，按优先级依次展开。

**① 专利工作流进度面板**

```rust
// crates/tui/src/widgets/patent_progress.rs
use ratatui::{
    buffer::Buffer,
    layout::Rect,
    style::{Color, Style},
    text::{Line, Span},
    widgets::{Block, Borders, List, ListItem, Widget},
};

#[derive(Debug, Clone, PartialEq)]
pub enum PatentStep {
    Pending,
    Running,
    Done,
    Error(String),
}

#[derive(Debug, Clone)]
pub struct PatentWorkflowStep {
    pub label: &'static str,
    pub state: PatentStep,
}

pub struct PatentProgressWidget<'a> {
    pub steps: &'a [PatentWorkflowStep],
    pub title: &'a str,
}

impl<'a> Widget for PatentProgressWidget<'a> {
    fn render(self, area: Rect, buf: &mut Buffer) {
        let items: Vec<ListItem> = self
            .steps
            .iter()
            .map(|step| {
                let (icon, style) = match &step.state {
                    PatentStep::Pending => ("○", Style::default().fg(Color::DarkGray)),
                    PatentStep::Running => ("◉", Style::default().fg(Color::Yellow)),
                    PatentStep::Done    => ("●", Style::default().fg(Color::Green)),
                    PatentStep::Error(_) => ("✗", Style::default().fg(Color::Red)),
                };
                ListItem::new(Line::from(vec![
                    Span::styled(format!("{icon} "), style),
                    Span::raw(step.label),
                ]))
            })
            .collect();

        List::new(items)
            .block(
                Block::default()
                    .borders(Borders::ALL)
                    .title(self.title)
                    .border_style(Style::default().fg(Color::DarkGray)),
            )
            .render(area, buf);
    }
}
```

首先，把 `packages/tui` 里已有的**业务逻辑**（不是渲染逻辑）迁移出来，合并进 `packages/cli` 或相关 Agent 包。渲染代码直接删除。


先把 DeepSeek-TUI 的 Engine/Agent 循环内部结构摸清楚。两边的执行引擎细节现在都清楚了。这是迄今为止最复杂的一个问题，先把两边的循环结构摆清楚再说融合。

------

## 先把两个循环的真实结构说透

**DeepSeek-TUI 的 Engine 循环**（来自 `core/engine.rs`）：

Agent Loop 位于 `core/engine.rs`，每条用户消息触发一个 turn：流式调用 LLM → 解析工具调用 → 经工具注册表执行 → 结果回写模型。`capacity_flow.rs` 提供防止上下文失控的护栏，压缩系统在上下文接近上限时智能摘要旧消息。

Sub-agents：模型可通过 `agent_spawn` 生成子 Agent，配合 `agent_wait`、`agent_result`、`agent_cancel` 完整生命周期管理。Hooks：pre/post 生命周期钩子，通过 `config.toml` 的 `[hooks]` 配置，支持 stdout/jsonl/webhook。RLM：内置递归 LM 工具，在沙箱中并行扇出 1–16 个 Flash 子任务。

当前已有的 Hook 事件：`tool_call_before`、`tool_call_after`、`message_submit`、`on_error`，以及 session 生命周期和模式切换事件。但 v0.8.8 中 Hooks 仍是只读观察者。

**YunPat 的 OrchestratorAgent**（来自 README）：位于 `packages/orchestrator`（17 个源文件），包含 `intent`/`planning`/`routing`/`exception` 四个子模块，是 TypeScript 实现的意图识别 → 任务规划 → 智能路由系统。

**核心矛盾**：DeepSeek-TUI 的 Engine 是一个**扁平的 ReAct 循环**（LLM → tool → LLM → tool → ...），而 YunPat 的 Orchestrator 是一个**分层的规划-执行架构**（意图识别 → 任务分解 → 子 Agent 路由 → 执行 → 异常处理）。这不是简单的接口对接，是两种不同的 Agent 设计哲学。

------

## 融合策略：不是替换，而是分层叠加

直接替换 Engine 循环是灾难性的——会丢失 DeepSeek-TUI 所有的会话管理、工具审批、上下文压缩等能力。正确的方案是**把 OrchestratorAgent 的能力作为一个"规划前置层"叠加在 Engine 循环之上**，通过三个接合点实现：

```
用户输入
    ↓
[接合点 A] pre-turn Hook → YunPat IntentRecognizer（意图识别）
    ↓
[接合点 B] message_submit Hook → YunPat TaskPlanner（任务规划注入）
    ↓
DeepSeek Engine 主循环（原样保留）
    ↓
[接合点 C] agent_spawn → YunPat 专利子 Agent（执行委托）
    ↓
结果回写 transcript
```

------

## 接合点 A：意图识别前置（`tool_call_before` Hook）

YunPat 的 `IntentRecognizer` 需要在每个 turn 开始前运行，判断用户意图属于哪类专利任务。DeepSeek-TUI 的 `message_submit` Hook 在用户消息提交时触发，正好是注入意图识别的位置。

在 `~/.deepseek/config.toml` 里配置 Hook（无需修改 Rust 源码）：

```toml
[hooks]
enabled = true

[[hooks.events]]
event   = "message_submit"
command = "node"
args    = [
  "/path/to/yunpat/packages/orchestrator-adapter/dist/intent-hook.js"
]
timeout = 5          # 意图识别必须在 5 秒内完成，否则 fallback
output  = "jsonl"    # Hook 输出 JSONL，Engine 读取结果
```

在 YunPat 侧实现这个 Hook 脚本。`orchestrator-adapter` 包（已有 4 个源文件，是 Rust 网关桥接）里新增：

```typescript
// packages/orchestrator-adapter/src/intent-hook.ts
// 作为 DeepSeek-TUI 的 message_submit Hook 运行
// 从 stdin 读取 Hook 上下文，向 stdout 输出 JSONL 指令

interface HookContext {
  event:    'message_submit'
  message:  string        // 用户输入
  session:  string        // session ID
  mode:     string        // plan/agent/yolo
}

// Hook 上下文从 stdin 读取
const raw  = await readStdin()
const ctx  = JSON.parse(raw) as HookContext

const recognizer = new IntentRecognizer()
const intent     = await recognizer.recognize(ctx.message)

// JSONL 输出：每行一条指令，DeepSeek-TUI 读取并执行

if (intent.type === 'patent_task') {

  console.log(JSON.stringify({

    mode:    'plan',

  console.log(JSON.stringify({

}

const patentSkillMap: Record<string, string> = {

}
```

## 接合点 B：任务规划注入（系统提示动态构建）

YunPat 的 `TaskPlanner` 负责把一个复杂专利任务分解为有序的子任务序列。这个能力不需要修改 Engine——通过 `instructions` 数组动态生成注入文件来实现。

在 YunPat 的 `orchestrator-adapter` 里新增一个"规划文件生成器"，在启动 DeepSeek-TUI 前调用：

```typescript
// packages/orchestrator-adapter/src/plan-injector.ts
// yunpat 启动脚本调用此函数，生成 .deepseek/current-plan.md
// 然后将该文件加入 config.toml 的 instructions 数组

import { TaskPlanner }      from '@yunpat/orchestrator/planning'
import { IntentRecognizer } from '@yunpat/orchestrator/intent'
import fs                   from 'node:fs/promises'
import path                 from 'node:path'

export async function injectPlanForSession(
  userRequest: string,
  workspaceDir: string,
): Promise<void> {
  const intent = await new IntentRecognizer().recognize(userRequest)

  if (intent.type !== 'patent_task') return   // 非专利任务不注入

  const plan   = await new TaskPlanner().plan(intent)
  const planMd = renderPlanAsMarkdown(plan)

  // 写入工作区级临时规划文件
  const planPath = path.join(workspaceDir, '.deepseek', 'current-plan.md')
  await fs.writeFile(planPath, planMd, 'utf-8')
}

function renderPlanAsMarkdown(plan: TaskPlan): string {
  return `# 当前专利任务规划
生成时间：${new Date().toLocaleString('zh-CN')}
任务类型：${plan.intentType}

## 执行步骤

${plan.steps.map((step, i) => `
### 步骤 ${i + 1}：${step.name}
- 工具：\`${step.tool}\`
- 输入：${step.inputDescription}
- 预期输出：${step.outputDescription}
- 前置条件：${step.prerequisites.join('、') || '无'}
`).join('\n')}

## 注意事项
${plan.constraints.map(c => `- ${c}`).join('\n')}

## 完成标准
${plan.completionCriteria}
`
}
```

把这个规划文件纳入 DeepSeek-TUI 的 `instructions` 数组（写入 `.deepseek/config.toml`）：

```toml
instructions = [
  ".deepseek/patent-system.md",
  ".deepseek/current-plan.md",   # 动态生成，任务开始前注入
  "AGENTS.md",
]
```

这样 DeepSeek-TUI 的 Engine 在执行每个 turn 时，system prompt 里已经包含了 YunPat TaskPlanner 生成的完整任务规划，Engine 的 LLM 调用天然会按照规划步骤推进，无需修改 Engine 代码。

## 接合点 C：专利子 Agent 委托（`agent_spawn` 工具）

这是最关键的接合点，也是两个系统真正"融合"而非"并列"的地方。

DeepSeek-TUI 的架构建议：把协调者（Orchestrator）和执行者（Worker）分开——主 session 保持小上下文，把独立任务委托给子 Agent，子 Agent 开启新的干净 session。

YunPat 的每个专利 Agent（Writer/Analyzer/Responder/Search）对应 DeepSeek-TUI 的一个 `agent_spawn` 调用。具体实现：在 YunPat 的 MCP server 里，除了直接返回结果的工具，还提供一组"异步委托型"工具，这类工具的响应不是立即返回结果，而是返回一个 `agent_spawn` 指令，让 DeepSeek Engine 自己去生成子 Agent：

```typescript
// packages/mcp-server/src/tools/async-patent-writer.ts
// 返回 agent_spawn 指令而不是直接执行

server.setRequestHandler(CallToolRequestSchema, async (request) => {
  if (request.params.name === 'patent_draft_async') {
    const { title, field, disclosure } = request.params.arguments

    // 不直接执行，而是生成一段让 DeepSeek Engine 去 spawn 子 Agent 的指令
    // 这样整个执行过程对用户可见，且受到 DeepSeek 的 session 管理保护
    return {
      content: [{
        type: 'text',
        text: `
请使用 agent_spawn 工具创建一个专利撰写子 Agent，配置如下：

agent_spawn({
  name: "patent-writer-${Date.now()}",
  task: "完整专利撰写任务",
  instructions: \`
    你是一名专利撰写专家。请完成以下专利申请文件的撰写：
    
    发明名称：${title}
    技术领域：${field}
    技术交底书：
    ${disclosure}
    
    执行步骤：
    1. 调用 mcp_yunpat_patent_search 检索现有技术
    2. 分析检索结果，确定权利要求保护范围
    3. 调用 mcp_yunpat_patent_draft_full 生成完整文件
    4. 检查权利要求与说明书的一致性
    5. 输出最终专利申请文件
  \`,
  model: "deepseek-v4-pro",
  mode: "agent",
})
`,
      }],
    }
  }
})
```

这个设计的好处是：专利撰写的整个过程在 DeepSeek-TUI 的 transcript 里完全可见，可以被 `/restore` 回滚，可以被 session 恢复，完全受 DeepSeek 的审批门控保护。

## 异常处理：OrchestratorAgent 的 Exception 模块对接

YunPat 的 `orchestrator/exception` 模块负责处理 Agent 执行失败、意图歧义、工具超时等异常。在 DeepSeek-TUI 侧，`on_error` Hook 事件在运行时错误时触发，可以把 YunPat 的异常处理逻辑接在这里：

```toml
# ~/.deepseek/config.toml 追加
[[hooks.events]]
event   = "on_error"
command = "node"
args    = [
  "/path/to/yunpat/packages/orchestrator-adapter/dist/exception-hook.js"
]
timeout = 10
output  = "jsonl"
// packages/orchestrator-adapter/src/exception-hook.ts
import { ExceptionHandler } from '@yunpat/orchestrator/exception'

interface ErrorContext {
  event:      'on_error'
  error_type: string
  error_msg:  string
  tool_name?: string
  turn_index: number
}

const ctx     = JSON.parse(await readStdin()) as ErrorContext
const handler = new ExceptionHandler()
const recovery = await handler.handle(ctx)

// 根据 YunPat 异常处理结果，输出恢复指令
switch (recovery.strategy) {
  case 'retry_with_context':
    // 告知 Engine 重试，并补充上下文
    console.log(JSON.stringify({
      action:  'inject_message',
      role:    'user',
      content: `上一步执行失败（${ctx.error_msg}）。${recovery.additionalContext}请重试。`,
    }))
    break

  case 'fallback_tool':
    // 建议换用备选工具
    console.log(JSON.stringify({
      action:  'suggest_tool',
      tool:    recovery.fallbackTool,
      reason:  recovery.reason,
    }))
    break

  case 'escalate_human':
    // 请求人工介入（触发 HITL）
    console.log(JSON.stringify({
      action:  'request_approval',
      message: `专利任务遇到需要人工判断的情况：${recovery.escalationReason}`,
      options: recovery.humanChoices,
    }))
    break
}
```

## 整体融合后的执行流

把上述三个接合点串联起来，一次完整的专利任务执行流如下：

```
用户：帮我撰写一件关于"基于Transformer的实时语音识别"的专利
         ↓
[A] message_submit Hook
    → IntentRecognizer：patent_writing (置信度 94%)
    → 切换 Plan 模式 + 注入专利上下文 + 加载 patent-writer Skill
         ↓
[B] TaskPlanner 生成 current-plan.md，注入 system prompt
    步骤1: 检索现有技术
    步骤2: 分析技术特征
    步骤3: 撰写权利要求
    步骤4: 撰写说明书
    步骤5: 生成摘要
         ↓
DeepSeek Engine 主循环（按规划步骤推进）
    → 调用 mcp_yunpat_patent_search
    → 分析检索结果（Engine 自主推理）
    → agent_spawn 委托 patent-writer 子 Agent
         ↓
[C] 子 Agent（patent-writer-session）
    → 调用 mcp_yunpat_patent_draft_full
    → 返回完整专利申请文件
         ↓
主 Agent 收到子 Agent 结果，整合呈现给用户
         ↓
[异常] 如有工具失败 → on_error Hook → ExceptionHandler → 恢复策略
```

这个设计的核心价值在于：YunPat 的专业能力（意图识别的准确性、任务规划的领域知识、异常处理的专利经验）完整保留，而 DeepSeek-TUI 的执行可靠性（会话管理、上下文控制、审批门控、workspace 回滚）也完整保留，两者通过 Hook + MCP + agent_spawn 三个接合点形成清晰的职责边界，而不是把两个循环硬塞在一起。
